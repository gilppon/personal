# 장애 대응 프로토콜

> 서비스 장애 발생 시 단계별 대응 절차

## 🚨 장애 등급 분류

| 등급 | 정의 | 대응 시간 | 예시 |
|------|------|----------|------|
| P0 | 서비스 전체 다운 | 즉시 | 서버 무응답, DB 접속 불가 |
| P1 | 핵심 기능 장애 | 1시간 내 | 결제 실패, 로그인 불가 |
| P2 | 부분 기능 장애 | 4시간 내 | 특정 페이지 에러, 느린 응답 |
| P3 | 경미한 이슈 | 다음 배포 | UI 깨짐, 오타 |

## 📋 P0/P1 대응 절차

### Phase 1: 감지 및 확인 (0~5분)
1. 장애 현상 정확히 기록 (에러 메시지, 스크린샷)
2. 영향 범위 파악 (전체 사용자? 특정 지역? 특정 기능?)
3. 최근 배포/변경 사항 확인

### Phase 2: 긴급 조치 (5~15분)
1. **최근 배포가 원인**: 즉시 롤백
2. **외부 서비스 장애**: 해당 서비스 상태 페이지 확인
3. **트래픽 급증**: Cloudflare Rate Limiting / 스케일링
4. **DB 이슈**: 커넥션 풀 확인, 슬로우 쿼리 kill

### Phase 3: 근본 원인 분석 (30분~)
1. 로그 분석 (에러 로그, 접속 로그)
2. 타임라인 정리 (언제 시작, 무엇이 트리거)
3. 재현 시도 (로컬 또는 스테이징)

### Phase 4: 영구 수정
1. 근본 원인에 대한 코드 수정
2. 테스트 → Preview 배포 → 검증
3. Production 배포

### Phase 5: 사후 분석 (Post-mortem)
```markdown
## 장애 보고서 — [날짜]

- **장애 시간**: HH:MM ~ HH:MM (총 X분)
- **영향 범위**: (사용자 수, 기능)
- **근본 원인**: (구체적 기술 원인)
- **조치 내용**: (무엇을 했는지)
- **재발 방지**: (어떤 조치를 추가할지)
- **교훈**: (팀/프로세스 차원의 배움)
```

## 🔍 일반적 장애 원인 TOP 5
1. **배포 실수**: 미테스트 코드 배포 → 자동 테스트 강화
2. **환경변수 누락**: 새 환경에 키 미설정 → 체크리스트
3. **DB 커넥션 고갈**: 커넥션 미반환 → 풀링/타임아웃 설정
4. **외부 API 장애**: 서드파티 서비스 다운 → 타임아웃 + 폴백
5. **트래픽 폭증**: 바이럴/DDoS → CDN 캐싱 + Rate Limit
