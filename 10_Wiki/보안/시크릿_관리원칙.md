# 시크릿 관리 원칙

> API 키, 비밀번호, 토큰 등 민감 정보의 안전한 관리

## 🔑 핵심 원칙

### 1. 코드와 시크릿은 반드시 분리
- 시크릿은 **환경변수**로만 관리
- 코드 저장소에 시크릿이 포함되면 즉시 유출된 것으로 간주
- 유출 시: 즉시 폐기 → 새 키 발급 → 연관 시스템 점검

### 2. 환경별 시크릿 분리
```
.env.local        → 로컬 개발용 (Git 제외)
.env.preview      → 스테이징/프리뷰용
.env.production   → 프로덕션용 (클라우드에서만 관리)
```

### 3. 클라이언트 vs 서버 구분
| 구분 | 접두사 예시 | 노출 범위 |
|------|-----------|----------|
| 서버 전용 | `DB_PASSWORD`, `API_SECRET` | 서버 코드에서만 접근 |
| 클라이언트 허용 | `NEXT_PUBLIC_*`, `VITE_*` | 브라우저에 노출됨 |

> ⚠️ DB 비밀번호, 결제 시크릿 키 등은 절대 `NEXT_PUBLIC_` 접두사 사용 금지

## 🛡️ 플랫폼별 시크릿 관리

### Cloudflare
- Workers: `wrangler secret put KEY_NAME`
- Pages: Dashboard → Settings → Environment Variables

### Supabase
- Row Level Security (RLS)로 클라이언트 직접 접근 제어
- Service Role Key는 서버에서만 사용

### GitHub
- Settings → Secrets and variables → Actions
- CI/CD에서만 접근 가능

## 📋 시크릿 관리 체크리스트
- [ ] `.gitignore`에 `.env*` 패턴 포함
- [ ] Git 히스토리에 시크릿 흔적 없음 (git log 확인)
- [ ] 프로덕션 시크릿은 로컬에 저장하지 않음
- [ ] 시크릿 로테이션 주기 설정 (90일 권장)
- [ ] 퇴사자/권한 변경 시 관련 키 즉시 교체
- [ ] 에이전트 `config.md` 파일은 `.gitignore` 처리

## 🚨 유출 대응 절차
1. **즉시**: 유출된 키 비활성화/삭제
2. **1시간 내**: 새 키 발급 및 배포
3. **당일**: 유출 경로 조사 (Git 히스토리, 로그 등)
4. **기록**: `_shared/decisions.md`에 인시던트 기록
