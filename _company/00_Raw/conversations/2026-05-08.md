### 1. 테이블 생성 (Schema Definition)

사용자 소유권 및 구독 계획 관리를 위해 두 개의 테이블을 정의합니다.

```sql
-- plans 테이블: 서비스 플랜 정의
CREATE TABLE plans (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL UNIQUE,
    description TEXT NOT NULL,
    price NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
    features JSONB  -- 기능 상세 정보를 저장할 JSONB 타입 사용
);

-- user_subscriptions 테이블: 사용자별 구독 상태 정의
CREATE TABLE user_subscriptions (
    id SERIAL PRIMARY KEY,
    user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE, -- Supabase Auth 연동을 가정
    plan_id INTEGER REFERENCES plans(id) ON DELETE RESTRICT,
    status VARCHAR(50) NOT NULL DEFAULT 'active', -- 예: active, trial, canceled
    start_date TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
    end_date TIMESTAMP WITH TIME ZONE
);

-- 인덱스 추가 (성능 최적화)
CREATE INDEX idx_user_subscriptions_user_id ON user_subscriptions(user_id);
CREATE INDEX idx_user_subscriptions_plan_id ON user_subscriptions(plan_id);
```

### 2. Row Level Security (RLS) 정책 적용

데이터 접근 권한을 사용자별로 엄격하게 제한하여 보안을 확보합니다. 기본적으로 인증된 사용자만 데이터에 접근할 수 있도록 설정합니다.

#### A. `plans` 테이블 RLS 정책

플랜 정보는 모든 사용자가 조회할 수 있도록 설정합니다 (공개 정보).

```sql
ALTER TABLE plans ENABLE ROW LEVEL SECURITY;

-- 모든 사용자가 플랜 정보를 읽을 수 있도록 허용
CREATE POLICY "Allow public read on plans" ON plans FOR SELECT USING (true);
```

#### B. `user_subscriptions` 테이블 RLS 정책

사용자는 자신의 구독 정보만 조회하거나 업데이트할 수 있도록 제한합니다.

```sql
ALTER TABLE user_subscriptions ENABLE ROW LEVEL SECURITY;

-- 1. 사용자 본인만 자신의 구독 정보를 읽을 수 있도록 허용
CREATE POLICY "Users can view their own subscriptions" ON user_subscriptions FOR SELECT USING (auth.uid() = user_id);

-- 2. 사용자 본인만 자신의 구독 상태를 업데이트할 수 있도록 허용
CREATE POLICY "Users can update their own subscription status" ON user_subscriptions FOR UPDATE USING (auth.uid() = user_id);

-- 3. 관리자(혹은 특정 역할)는 모든 구독 정보를 조회할 수 있도록 설정 (선택 사항, 현재는 기본 사용자만 접근 허용)
```

### 2. Row Level Security (RLS) 정책 적용

데이터 접근 권한을 사용자별로 엄격하게 제한하여 보안을 확보합니다. 기본적으로 인증된 사용자만 데이터에 접근할 수 있도록 설정합니다.

```sql
ALTER TABLE plans ENABLE ROW LEVEL SECURITY;

-- 모든 사용자가 플랜 정보를 읽을 수 있도록 허용
CREATE POLICY "Allow public read on plans" ON plans FOR SELECT USING (true);
```

#### B. `user_subscriptions` 테이블 RLS 정책

```sql
ALTER TABLE user_subscriptions ENABLE ROW LEVEL SECURITY;

-- 1. 사용자 본인만 자신의 구독 정보를 읽을 수 있도록 허용
CREATE POLICY "Users can view their own subscriptions" ON user_subscriptions FOR SELECT USING (auth.uid() = user_id);

-- 2. 사용자 본인만 자신의 구독 상태를 업데이트할 수 있도록 허용
CREATE POLICY "Users can update their own subscription status" ON user_subscriptions FOR UPDATE USING (auth.uid() = user_id);
```

## 최종 랜딩 페이지 UI/UX 명세서 및 와이어프레임 확정 (3-Tier 전략 기반)

본 문서는 Business의 3-Tier 가격 전략(Starter, Accelerator, Scale)과 핵심 가치 제안을 Developer의 기술적 구현 가능성과 결합하여, 사용자 전환율을 극대화하는 최종 랜딩 페이지의 UI/UX 명세와 와이어프레임을 확정합니다.

### 1. 디자인 컨셉 요약 (Design Philosophy)

**핵심 원칙:**
*   **Spatial-First & Depth:** 단순한 정보 나열이 아닌, 각 티어(Tier)가 물리적인 공간 속에서 위계적으로 분리되어 보이며 사용자가 다음 단계로 이동하도록 유도하는 깊이감 있는 레이아웃을 적용한다. (Z-Axis Composition 활용)
*   **Tactile Texture:** 데이터와 가치를 시각적 질감으로 표현하여, 가격표가 단순한 숫자가 아닌 '투자 대비 경험의 차이'로 느껴지게 한다.
*   **Clarity & Directness:** 직설적인 톤을 유지하되, 각 티어 간의 차이를 명확하게 인지시키고 다음 단계로의 흐름(Flow)을 즉각적으로 제공한다.

**컬러 및 타이포그래피 적용 (HSL 기반 설계):**
*   **Primary Color (넥스트하루 브랜드):** 짙은 네이비 계열 (Deep Navy, H=220, S=10%, L=35%) — 신뢰감과 전문성 확보.
*   **Accent Color (CTA & Highlight):** 차분한 민트/세이지 계열 (Sage Green, H=135, S=15%, L=45%) — 성장, 긍정적 변화의 느낌 부여.
*   **Background:** 미묘한 노이즈 오버레이 또는 메쉬 그라디언트를 적용하여 평면성을 피하고 촉각적인 질감을 부여한다.

### 2. 와이어프레임 상세 (Wireframe Flow)

페이지는 사용자의 인지 부하를 최소화하고, 가장 적합한 경로로 안내하는 **'선택 유도형 계단식 구조(Stepped Hierarchy)'**를 채택합니다.

| 섹션 | 제목/목표 | 핵심 내용 및 UI/UX 요소 | CTA 배치 전략 |

| **A. Hero Section (상단)** | **문제 정의 & 비전 제시** | **H1:** "더 나은 생활을 위한 [서비스명] 솔루션" (직설적 가치 선언). 강력한 헤드라인과 배경의 깊이감(Depth) 강조. | 메인 CTA: **'무료 진단 시작하기'** (가장 눈에 띄는 Accent Color 사용). |

### 3. CTA 배치 최적화 전략 (Optimization Strategy)

CTA는 사용자의 **인지 수준**과 **심리적 부담감**에 따라 다르게 설정하여 전환율을 높입니다.

1.  **최상단 CTA (Hero):** **'무료 진단 시작하기'**
    *   **목표:** 낮은 장벽으로 유입 확보 (정보 탐색 단계).
    *   **배치:** 페이지 로딩 후 즉시 시선이 가는 중앙 상단에 배치.
2.  **중간 CTA (Tier Comparison):** **각 티어별 맞춤형 CTA**
    *   **목표:** 사용자가 자신의 니즈(Starter, Accelerator, Scale)를 인지하는 순간 행동을 유도 (결정 단계).
    *   **전략:** 각 가격 플랜 카드 하단에 명확히 구분된 CTA를 배치하여, 사용자가 자신이 원하는 목표에 맞는 옵션을 선택하도록 강제한다. 이는 **'선택의 고통'**을 최소화하고 다음 행동으로 자연스럽게 연결시킨다.
3.  **최하단 CTA (Footer):** **'플랜 선택하기' 또는 '지금 시작하기'**
    *   **목표:** 모든 정보를 검토한 사용자를 최종 전환 단계로 이끌기.
    *   **배치:** 페이지의 가장 아래에 위치시키되, 상단의 CTA와는 시각적으로 분리하여 명확하게 마무리한다.

## ✨ 핵심 시각 자료 사양 (Visual Material Specification)

이 사양은 **2026 Spatial Design Standards**와 **UI/UX 핵심 원칙**을 완벽하게 준수하며, 개발 단계에서 바로 적용 가능하도록 구체화되었습니다.

### 1. 색상 팔레트 (HSL Color Engineering)

모든 색상은 순색(Pure) 사용을 금지하고, HSL 기반으로 촉각적 질감과 공간감을 동시에 부여합니다.

| 역할 | HSL 값 (Base) | HEX 변환 예시 (톤 조절됨) | 적용 범위 및 용도 |
| :--- | :--- | :--- | :--- |
| **Primary (Accent)** | H: 240, S: 80%, L: 60% | `#638C9E` (차분한 청회색) | 주요 CTA 버튼, Accelerator 강조 요소. 공간감(Blur/Shadow)의 기준 색상. |
| **Secondary (Background)** | H: 210, S: 10%, L: 95% | `#E6F2FF` (매우 밝은 하늘색) | 메인 배경 영역. 촉각적 질감(Texture) 오버레이의 기반. |
| **Tertiary (Text/Contrast)** | H: 240, S: 10%, L: 30% | `#3A4D59` (짙은 회색-블루 계열) | 본문 텍스트 및 섹션 제목. 명확한 가독성 확보. |
| **Shadow/Depth** | H: 240, S: 10%, L: 15% | `#B1C7D8` (밝은 회색-블루) | Z-Axis 깊이감 및 다층 그림자(Multi-layered shadow)의 베이스. |

### 2. 타이포그래피 시스템 (Typography System)

모바일 퍼스트 원칙을 준수하며, 명확한 계층 구조를 확립합니다.

*   **시스템 폰트:** Pretendard (한글), Outfit 또는 Inter (영문).
*   **8px 그리드 적용:** 모든 여백(Padding/Margin)은 **8의 배수**로 설계합니다. (예: `padding: 32px` = 4 * 8n)
*   **헤드라인 계층:** H1 (Hero), H2 (섹션 제목), H3 (카드 제목), Body Text, Caption 순으로 명확히 분리하며, 각 레벨 간의 **Line Height는 최소 1.6**을 유지하여 시각적 여유를 확보합니다.

### 3. Z-Axis 및 질감 적용 지침

*   **깊이감 구현:** 모든 주요 카드(Pricing Tier)와 Hero 섹션에는 `backdrop-filter: blur(12px)`와 미묘한 **다층 그림자(Multi-layered Shadow)**를 적용하여 요소가 배경 위로 떠 있는 듯한 '공간감'을 극대화합니다.
*   **촉각적 질감:** 배경색(`Secondary`) 위에 매우 낮은 불투명도(`opacity: 0.05` ~ `0.1`)의 노이즈 오버레이 또는 메쉬 그라디언트 레이어를 적용하여 단조로움을 피하고 질감을 부여합니다.
*   **애니메이션:** 모든 인터랙션(Hover, Click)에는 **0.3s ~ 0.4s**의 `cubic-bezier` 트랜지션을 적용하여 반응 속도를 높이고 리듬감을 제공합니다.

| 기능 영역 | HTTP Method | Endpoint Path | 설명 (Purpose) | 필요한 데이터 소스 | 보안/RLS 요구사항 |

### 3. 공간 및 레이아웃 규칙 (Spatial & Grid Rules)

모든 여백(Padding/Margin)과 크기는 **8의 배수**로 설계되어 일관된 리듬감을 부여합니다. Z-Axis 효과는 `backdrop-filter`와 다층 그림자를 통해 구현됩니다.

#### A. 그리드 및 간격 (Grid & Spacing)
*   **기본 단위:** `8px` (모든 Padding, Margin, Border의 기본 단위)
*   **Spacing Scale:** 모든 간격은 8의 배수로 설정합니다. (예: `16px`, `24px`, `32px`, `40px` 등)

#### B. Z-Axis 및 깊이 구현 (Depth Implementation)
요소들이 물리적 공간에 부유하는 느낌을 주기 위해 다층 그림자(Multi-layered shadow)를 필수적으로 사용합니다.

**핵심 CSS 스타일 정의:**

```css
/* 1. 기본 컨테이너 스타일: 모든 요소의 배경은 질감을 갖도록 설정 */
.element {
    background-color: var(--color-bg-base);
    /* Tactile Texture 적용 예시 (Mesh Gradient 또는 Noise Overlay) */
    /* background-image: url('noise_texture.png'); /* 실제 이미지 경로로 대체 */
    /* background-blend-mode: multiply; */
}

/* 2. 깊이 표현을 위한 다층 그림자 (Z-Axis Effect) */
.layered-card {
    /* 기본 배경 레이어 (가장 뒤쪽, 공간감 부여) */
    box-shadow: 0 12px 30px rgba(0, 0, 0, 0.05), /* 깊은 그림자 */
                0 6px 18px rgba(0, 0, 0, 0.03); /* 중간 그림자 */
    /* Z-Axis 효과를 위한 배경 블러 (물리적 공간 부유) */
    backdrop-filter: blur(15px); /* 핵심: 뒤 배경을 흐리게 처리하여 깊이감 생성 */
}

/* 3. 상호작용 애니메이션 (Micro-Animation 적용) */
a, button {
    transition: all 0.3s cubic-bezier(0.4, 0, 0.2, 1); /* 리듬감 있는 피드백 제공 */
}

/* 4. CTA 버튼 스타일 (Accent Color 활용) */
.cta-button {
    background-color: var(--color-accent-primary);
    color: var(--color-text-primary);
    border: none;
    padding: var(--spacing-medium, 24px) var(--spacing-large, 32px); /* 8배수 적용 */
    box-shadow: 0 10px 20px rgba(33, 125, 255, 0.4); /* 액센트 색상 기반 그림자 */
}
```

## 📊 심리학 기반 메시지 프레이밍 분석 보고서 (Pain vs. Gain)

본 보고서는 '고통 강조(PAS)'와 '결과 제시(Outcome Focus/Gain Framing)'의 전환율에 대한 행동경제학 및 마케팅 심리학적 연구 결과를 요약하고, 넥스트하루의 타깃 청중(개발자/1인 기업가)에게 가장 효과적인 메시지 구조를 제안합니다.

### I. 핵심 이론 요약: 손실 회피 편향 (Loss Aversion Bias)

| 구분 | 내용 | 심리학적 근거 | 적용 원칙 |
| :--- | :--- | :--- | :--- |
| **개념** | 인간은 동일한 크기의 이득(Gain)을 얻었을 때 느끼는 만족감보다, 손실(Loss)을 경험했을 때 느끼는 고통에 약 2배 이상의 감정적 반응과 심리적 중요성을 부여함. | **전망 이론 (Prospect Theory)** - Daniel Kahneman & Amos Tversky. 인간의 의사결정은 이득 최대화가 아닌, 손실 최소화를 목표로 함. | 메시지 설계 시, '무엇을 얻게 될까?'보다 **'무엇을 잃지 않을 수 있을까?'**에 초점을 맞춘다. |
| **적용** | 개발자의 시간과 자원 낭비(오류), 시장 진출 기회 상실(홍보 지식 부족) 등 현재의 '위험'을 극대화하여 고통으로 제시해야 구매 욕구가 높아짐. |

### II. 메시지 프레이밍 비교 분석 (데이터 기반)

| 구분 | Pain Amplification (PAS) | Outcome Focus (Gain Framing) | 효율성 및 추천도 |
| :--- | :--- | :--- | :--- |
| **심리적 자극** | 공포, 불안, 위협감 등 부정적인 감정 유발. | 희망, 기쁨, 성취감 등 긍정적인 기대감 조성. | **PAS (강력 추천)** |
| **작동 원리** | 현재 상태의 문제점을 구체적 손실(시간/돈)로 증폭시켜 *탈출의 필요성*을 느끼게 함. | 목표 달성을 상상하게 하여 *구매 동기*를 부여함. |
| **타깃 반응 (개발자)** | "내가 지금 이렇게 비효율적으로 일하고 있다는 사실"에 즉각적인 불편함을 느낌 → 해결책에 민감도 극대화. | "나중에 성공할 수 있을 거야"라는 추상적 기대는 행동으로 이어지지 않음. |
| **문제점** | 지나치게 과장되거나 공포심을 조장할 경우, 브랜드 신뢰도가 하락할 위험이 있음. (⚠️ 적절한 톤앤매너 유지 필수) | 모호하고 일반적이어서 경쟁사들과 차별화가 어려움. |

### III. [결론] 최적의 메시지 구조 제안: '손실-위험 강조형 PAS'

넥스트하루의 상황(최소 비용 고부가 가치 웹앱 제작 + 홍보 지식 부족)을 고려할 때, **‘기술적 비효율성’**과 **‘시장 진출 실패 리스크’**라는 두 가지 핵심 손실에 초점을 맞춘 PAS 구조가 가장 강력합니다.

| 단계 | 목표 (What to Say) | 적용 예시 (개발자 대상) | 목적 달성 기여도 |
| :--- | :--- | :--- | :--- |

### 1. HSL 기반 Semantic Color Palette (HSL Color Engineering)

| **Text Primary** (핵심 메시지) | Off-White (가독성 극대화) | `hsl(0, 0%, 95%)` | 모든 핵심 헤드라인 및 CTA 텍스트. 대비를 최대화하여 가독성을 보장합니다. |

*   **표면 (Surface):** 미세한 내부 그림자 (Soft, Low Opacity)
*   **부유감 (Depth):** 외곽에 넓고 흐릿한 블러드(Blur) 효과를 적용하여 요소가 배경으로부터 분리되어 부유하는 느낌을 구현합니다. (`backdrop-filter: blur(12px);` 권장)

### 1. 최종 API 엔드포인트 및 데이터 매핑 재확인 (Backend Data Mapping Confirmation)

기존에 확정된 Supabase 스키마(`plans`, `user_subscriptions`)와 Designer의 Z-Axis 흐름을 기반으로, 프론트엔드가 필요로 하는 핵심 API 엔드포인트와 데이터 매핑을 최종적으로 정리합니다.

**핵심 데이터 모델:**
*   `plans`: 구독 플랜 정보 (Starter, Accelerator, Scale) 및 각 플랜에 따른 가격/혜택 정의.
*   `user_subscriptions`: 사용자별 현재 구독 상태 및 활성화된 플랜 정보.

| 기능 요구사항 | 필요한 API 엔드포인트 | 데이터 매핑 (Supabase Column -> Frontend State) | 비고 (보안/성능) |
| :--- | :--- | :--- | :--- |
| **플랜 목록 조회** | `GET /api/plans` | `plans` 테이블 전체 (ID, Name, Price, Description, Features) | RLS를 통해 사용자에게만 접근 허용. |
| **사용자 구독 정보** | `GET /api/user-subscription` | `user_subscriptions`에서 현재 사용자의 활성 플랜 데이터 매핑 | 세션 인증(JWT) 기반으로 권한 검사 필수. |
| **구독 플랜 선택** | `POST /api/subscribe` | `plans` 테이블의 특정 Plan ID와 사용자 정보를 삽입 (트랜잭션 보장). | RLS 및 DB 트랜잭션을 통해 데이터 무결성 확보. |

**개발 준비사항:**
1.  **RLS 정책 검토**: `user_subscriptions` 테이블에 대해 `auth.uid()`를 사용하여 본인만 접근 가능하도록 RLS 정책을 철저히 적용합니다. (보안-by-디자인 원칙 준수)
2.  **API 핸들러 설계**: Next.js의 Route Handler(`app/api/...`) 내에서 Supabase 클라이언트(PostgREST 또는 직접 연결)를 사용하여 비즈니스 로직을 처리하고, 모든 데이터 접근은 인증된 사용자 ID에 기반해야 합니다.

### 2. 초기 프론트엔드 구조 (Next.js Skeleton) 준비

Designer가 제시한 Z-Axis 흐름과 HSL 스타일링 가이드라인을 반영하여, 핵심 섹션(Hero, Pain & Agitate, Solution, Feature, Pricing/CTA)을 포함하는 기본 레이아웃 뼈대를 Next.js App Router 기준으로 정의합니다.

**파일 구조 제안:**
```
app/
└── page.tsx  <-- 메인 랜딩 페이지
    ├── layout.tsx <-- 전역 스타일 및 컨텍스트 설정 (Z-Axis 변수 주입)
    └── sections/
        ├── Hero.tsx          <-- 1. Entry Point
        ├── PainAgitate.tsx   <-- 2. Ascend (깊이감 강조 섹션)
        ├── Solution.tsx      <-- 3. Pivot (전환점 섹션)
        ├── Features.tsx      <-- 4. Explore (좌우 분할 레이아웃)
        └── PricingCTA.tsx    <-- 5. Exit Point (Z-Axis 강조 CTA)
```

**`app/page.tsx` 기본 구조 (JSX Skeleton):**

```tsx
// app/page.tsx

import { HeroSection } from './sections/Hero';
import { PainAgitateSection } from './sections/PainAgitate';
import { SolutionSection } from './sections/Solution';
import { FeaturesSection } from './sections/Features';
import { PricingCTASection } from './sections/PricingCTA';

// Z-Axis Flow를 반영한 메인 컴포넌트
export default function LandingPage() {
  return (
    <main className="min-h-screen bg-gray-50 relative overflow-hidden">
      {/* 1. Hero Section: Entry Point */}
      <HeroSection />

      {/* 2. Pain & Agitate: Ascend - 깊이감 강조 영역 */}
      <PainAgitateSection />

      {/* 3. Solution Introduction: Pivot - 시각적 전환점 */}
      <SolutionSection />

      {/* 4. Feature & Value: Explore - 좌우 분할 레이아웃 */}
      <FeaturesSection />

      {/* 5. Pricing & CTA: Exit Point - 최종 행동 유도 */}
      <PricingCTASection />

      {/* 배경에 미묘한 Z-Axis 효과를 위한 오버레이 또는 패턴 추가 가능 (CSS/Tailwind 활용) */}
    </main>
  );
}

// 참고: layout.tsx에서 HSL 및 Z-Axis 변수를 전역으로 정의하여 모든 컴포넌트가 일관된 공간감을 갖도록 설정해야 합니다.
```

프론트엔드 구현을 위해, 저는 이제 `sections/Hero.tsx`부터 순차적으로 디자인 사양에 맞추어 컴포넌트를 상세히 구현하는 작업을 진행하겠습니다.
