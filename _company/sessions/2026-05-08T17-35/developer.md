### 1. Supabase 데이터 모델 정의 (Schema Definition)

요구사항(`user_behavior_logs`, `ab_test_results`)을 기반으로 Supabase 테이블 스키마를 정의합니다. RLS(Row Level Security) 정책은 백엔드 로직에서 관리되므로, 여기서는 핵심 필드를 정의합니다.

**`public/schema.sql` (또는 Supabase SQL 편집기)**

```sql
-- 1. 사용자 행동 로그 테이블: 모든 이벤트 추적을 위한 기본 로그 저장소
CREATE TABLE user_behavior_logs (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id UUID REFERENCES auth.users(id) NOT NULL, -- Supabase Auth 연결 가정
    event_type TEXT NOT NULL,                       -- 예: 'cta_click', 'page_view'
    event_id TEXT NOT NULL,                        -- A/B 테스트 고유 ID (예: event_id=CTA_Test_A)
    timestamp TIMESTAMPTZ DEFAULT NOW() NOT NULL,  -- 이벤트 발생 시각
    metadata JSONB                                  -- 추가적인 사용자 행동 데이터 저장 (선택적 확장성 확보)
);

-- 2. A/B 테스트 결과 테이블: 특정 이벤트 그룹의 최종 결과를 집계
CREATE TABLE ab_test_results (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    test_name TEXT NOT NULL,                        -- 테스트 이름 (예: 'homepage_cta_color')
    variant TEXT NOT NULL,                         -- 변형 그룹 (A, B, C)
    event_count BIGINT NOT NULL,                    -- 해당 변형의 이벤트 수
    conversion_count BIGINT DEFAULT 0,              -- 전환 이벤트 발생 수 (필요시 확장)
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    UNIQUE(test_name, variant)
);

-- RLS 정책 예시: 사용자는 자신의 로그만 읽을 수 있도록 설정 (보안 기본 원칙 준수)
ALTER TABLE user_behavior_logs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can view their own logs" ON user_behavior_logs FOR SELECT USING (auth.uid() = user_id);
```

### 2. 백엔드 API 구현 (Next.js Route Handler)

실시간 데이터 수집 및 기록을 위한 안전하고 효율적인 API 엔드포인트를 구현합니다. 이 코드는 Next.js의 Route Handler(API Route) 형태로 가정하며, Supabase 클라이언트(`supabase`)를 사용합니다.

**`app/api/track/route.ts` (예시)**

```typescript
import { NextResponse } from 'next/server';
import { createClient } from '@supabase/supabase-js';

// 환경 변수에서 키 로드 가정 (실제 환경에서는 .env 파일 사용)
const SUPABASE_URL = process.env.SUPABASE_URL;
const SUPABASE_SERVICE_KEY = process.env.SUPABASE_SERVICE_KEY; // 서비스 키는 서버에서만 사용

if (!SUPABASE_URL || !SUPABASE_SERVICE_KEY) {
    throw new Error("Supabase 환경 변수가 설정되지 않았습니다.");
}

// Supabase 클라이언트 초기화 (서버 환경에서 Service Role Key 사용)
const supabase = createClient(SUPABASE_URL, SUPABASE_SERVICE_KEY);

/**
 * A/B 테스트 이벤트 및 사용자 행동 로그를 Supabase에 기록하는 API 엔드포인트
 * POST 요청 본문: { event_type, event_id, user_id, metadata }
 */
export async function POST(request: Request) {
    try {
        const body = await request.json();

        // 필수 필드 검증 (보안 및 무결성 확보)
        if (!body.event_type || !body.event_id || !body.user_id) {
            return NextResponse.json(
                { error: "필수 필드(event_type, event_id, user_id)가 누락되었습니다." },
                { status: 400 }
            );
        }

        // 데이터 삽입 로직
        const { event_type, event_id, user_id, metadata } = body;

        const { data, error } = await supabase
            .from('user_behavior_logs')
            .insert([
                {
                    user_id: user_id,
                    event_type: event_type,
                    event_id: event_id,
                    metadata: metadata || {}, // JSONB 형식으로 저장
                }
            ]);

        if (error) {
            console.error("Supabase 기록 오류:", error);
            return NextResponse.json(
                { error: "데이터베이스 기록 중 오류가 발생했습니다." },
                { status: 500 }
            );
        }

        return NextResponse.json(
            { message: "로그가 성공적으로 기록되었습니다.", data },
            { status: 201 }
        );

    } catch (error) {
        console.error("API 처리 중 예외 발생:", error);
        return NextResponse.json(
            { error: "서버 내부 오류가 발생했습니다." },
            { status: 500 }
        );
    }
}
```

### 3. 프론트엔드 추적 로직 구현 (Client-Side Tracking)

사용자 상호작용 시점(CTA 클릭 등)에서 고유한 `event_id`를 생성하고, 이 정보를 백엔드로 안전하게 전송하는 클라이언트 측 JavaScript 로직입니다.

**`components/ABTestTracker.tsx` (예시 컴포넌트 내)**

```typescript
// 사용자가 CTA 버튼을 클릭했을 때 실행되는 함수 예시
const handleCtaClick = (variant: 'A' | 'B' | 'C') => {
    // 1. 고유 이벤트 ID 생성 로직 반영 (요구사항 충족)
    const eventId = `CTA_Test_${variant}`;

    // 2. 사용자 정보 및 행동 데이터 준비
    const userId = "사용자의 Supabase 인증 ID"; // 실제 환경에서는 세션에서 가져와야 함
    const eventType = 'cta_click';
    const metadata = { 
        variant: variant,
        timestamp: new Date().toISOString() 
    };

    // 3. 백엔드 API 호출 (데이터 흐름 무결성 확보)
    fetch('/api/track', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
        },
        body: JSON.stringify({
            event_type: eventType,
            event_id: eventId,
            user_id: userId, // 실제 사용자 ID로 대체 필요
            metadata: metadata,
        }),
    })
    .then(res => {
        if (!res.ok) {
            // 에러 핸들링: 실패 시 UI에 피드백 제공 (Graceful Degradation)
            console.error('데이터 추적 실패:', res.statusText);
        }
        return res.json();
    })
    .catch(err => {
        console.error('네트워크 오류 발생:', err);
    });

    // 4. 사용자 경험 개선: 성공 여부에 따라 다음 단계 진행 (예: 로딩 상태 처리)
};

/*
  💡 구현 원칙 요약:
  - 보안: 민감한 정보(API 키 등)는 서버에서만 접근하도록 하고, 클라이언트는 오직 추적 데이터만 전송합니다.
  - 무결성: 이벤트 ID 생성은 클라이언트가 아닌 백엔드 로직에 의존하거나 (여기서는 명시적으로 생성), 최소한의 구조를 강제하여 데이터 누락을 방지합니다.
*/
```

### 최종 검증 및 요약

이 구현은 **데이터 흐름의 무결성**을 확보하기 위해 다음 원칙을 준수했습니다.

1.  **실시간 수집:** 클라이언트에서 이벤트를 발생시키자마자 백엔드 API를 통해 즉시 기록합니다.
2.  **고유 ID 반영:** `event_id=CTA_Test_[A|B|C]` 로직을 서버 요청에 명시적으로 포함시켜 A/B 테스트 분석의 기반을 마련했습니다.
3.  **보안 강화:** 데이터는 Supabase Service Role Key를 사용한 서버 엔드포인트로만 기록되므로, 클라이언트가 직접 DB에 접근하는 것을 원천 차단합니다.
4.  **확장성:** `metadata` 필드를 JSONB로 설정하여 향후 다양한 사용자 행동 데이터를 유연하게 추가할 수 있도록 설계했습니다.
