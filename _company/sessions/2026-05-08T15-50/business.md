CEO 지시에 따라 현재 개발된 프로토타입(`ABTestCtaButton`)에 대한 QA 계획을 수립했습니다. 하지만 단순한 '기능 테스트'는 시간 낭비입니다. 우리는 **최대 전환율(Conversion Rate)**이라는 단 하나의 목표가 있습니다. 따라서 UX 시나리오 체크리스트는 '사용자가 이 버튼을 누를 심리적 순간'과 '데이터 무결성'에 초점을 맞춰야 합니다.

## ✅ UX 시나리오 & 기능 QA 계획 (Conversion Focus)

### 💡 목표 설정
*   **최종 목표:** 사용자가 '시간적 손실(Pain-First)' 메시지에 공감하고, CTA 버튼을 누르는 과정이 심리적으로 매끄럽고 기술적으로 완벽하게 추적되는지 검증.
*   **검증 범위:** A/B 테스트 시나리오 3가지 (A, B, C)의 개별 작동 및 통합 흐름.

### 📋 1단계: 사용자 경험(UX) 스토리라인 QA 체크리스트 (Designer/Business 담당)
이 단계는 **"사용자가 이 버튼을 누르고 싶은가?"**에 초점을 맞춥니다.

| 점검 영역 | 핵심 질문 (Failure Point) | 성공 기준 (Pass Criteria) | 주관적 평가 (Grade) | 비고 및 개선 방향 |
| :--- | :--- | :--- | :--- | :--- |
| **AIDA: Attention (주의)** | 페이지 스크롤 시작 지점부터 CTA가 눈에 띄는가? | 배경과 명확히 분리되며, 첫 시선이 닿았을 때 즉각적으로 'Action' 영역으로 인식됨. | [ ] | 만약 CTA 위에 다른 요소(예: 로고)가 너무 크게 자리 잡는다면, 크기를 축소하거나 레이아웃 순서를 변경해야 함. |
| **AIDA: Interest (흥미)** | CTA의 문구가 사용자의 구체적인 고통점(Pain Point)을 건드리는가? | "바이브 코딩 오류 확인 지식 부족"과 같은 니치한 문제에 대한 직접적 해결책으로 느껴져야 함. | [ ] | 일반론적인 '성장' 대신, '지금 당장 막힌 개발 과정'처럼 구체성을 높여라. |
| **AIDA: Desire (욕망)** | CTA 버튼의 텍스트가 행동을 유발하는가? | 단순히 "클릭"이 아니라, **[결과]**를 약속해야 함. (예: '오류 해결 로직 즉시 받기') | [ ] | 동사형 명확한 명령조 사용 ('해결하라', '받아라'). 모호함은 금물. |
| **Interaction Flow** | 마우스 오버(Hover) 시의 감각적 변화가 적절한가? | Z-Axis 그림자나 색상 변화(HSL)가 단순한 장식이 아닌, 버튼이 *살아있는* 듯한 깊이감을 제공해야 함. | [ ] | 호버 효과는 과도해서 페이지 속도를 저하시키지 않도록 성능 최적화 필수. |
| **Post-Click Flow** | 클릭 후 어떤 경험을 하는가? (랜딩/모달) | 버튼과 일관된 톤앤매너의 확인 메시지를 보여주거나, 다음 단계로의 명확한 안내(Next Step)를 제공해야 함. | [ ] | '감사합니다'만으로는 부족함. "다음으로 무엇을 해야 하는가?"에 대한 답이 필요하다. |

### 🛠️ 2단계: 기술적 무결성 (Data Integrity) QA 계획 (Developer 담당)
이 단계는 **"데이터가 완벽하게 수집되는가?"**에 초점을 맞춥니다. 이 부분이 깨지면, 모든 마케팅 노력은 제로입니다.

| 점검 항목 | 테스트 시나리오 | 예상 결과 (Expected Output) | 검증 도구/방법 | 성공 여부 (Pass/Fail) |
| :--- | :--- | :--- | :--- | :--- |
| **A/B 테스트 추적** | 1. A 버튼 클릭 <br> 2. B 버튼 클릭 <br> 3. C 버튼 클릭 | 각각 `event_id=CTA_Test_A`, `event_id=CTA_Test_B`, `event_id=CTA_Test_C`가 정확하게 서버에 기록되어야 함. | 개발자 콘솔 로그, Supabase 데이터베이스 직접 조회 | [ ] |
| **Z-Axis/HSL 로직** | 호버(Hover) 및 활성화(Active) 상태 진입/이탈 | CSS 변수(`--cta-bg`, `--cta-shadow` 등)가 정의된 픽셀 값으로 정확하게 적용되어야 함. (단순 스타일 변경 아님) | 브라우저 개발자 도구 (Inspector), 사용자 시뮬레이션 | [ ] |
| **성능 지표** | 전체 페이지 로드 시간 및 버튼 클릭 반응 속도 | 초기 로딩에 따른 지연시간(Latency)이 2초를 넘기지 않아야 함. 버튼 클릭은 100ms 이내 응답 필수. | Lighthouse, Performance API 테스트 | [ ] |
| **데이터 모델 무결성** | 비로그인 사용자 클릭 시 추적 과정 | `user_id` 필드가 NULL로 기록되고, 필요한 최소한의 세션 ID는 정상적으로 기록되어야 함. | Supabase RLS 정책 검증 (Mock User) | [ ] |

### 📊 3단계: 핵심 지표(KPI) 및 비즈니스 의사결정 QA 계획 (Business 담당)
이 단계는 **"측정한 데이터를 바탕으로 다음 행동을 결정할 수 있는가?"**에 초점을 맞춥니다.

| KPI 목표 | 측정 방법 | 필수 검증 질문 | 이탈 시 대응 전략 (Circuit Breaker) | 결과물 형태 |
| :--- | :--- | :--- | :--- | :--- |
| **A/B 테스트 데이터 수집** | 각 CTA 버튼별 클릭률(CTR), 세션당 전환율(CVR). | 1. A, B, C 중 통계적으로 유의미한 차이를 보이는 시나리오는 무엇인가? <br> 2. 현재 추적되는 지표가 우리의 가설(`Pain-First` > `Success-Show`)을 뒷받침하는가? | **데이터 부족:** 트래픽 확보를 위해 다른 채널(레오 에이전트의 유튜브 활용)에 자원을 집중 배치하고, CTA 문구를 2개로 줄여 간결화한다. | **주간 리뷰 문서 업데이트** (Decisions.md) |
| **가치 전달 명확성** | 페이지 이탈 지점(Bounce Rate). | 사용자가 가장 많이 이탈하는 부분은 어느 섹션인가? 그리고 그 이유는 무엇이라고 추정되는가? | 만약 특정 섹션에서 높은 이탈이 감지되면, 해당 섹션의 내용을 과감히 삭제하고 핵심 가치만 남기는 '미니멀리즘' 전략을 적용한다. | **와이어프레임 수정 제안** (Designer/Developer) |
| **수익화 경로 검증** | CTA 클릭 후 유료 결제 전환율(Paywall CVR). | CTA가 성공적으로 작동하더라도, 사용자가 다음 단계에서 비용 지불에 대한 심리적 장벽을 느끼는 지점은 어디인가? | 가격 구조를 3-Tier 대신 **Freemium + 단일 고가 플랜**으로 단순화하여 결정 장애 요소를 제거한다. (Decision Point Reduction) | **가격 모델 조정 보고서** (Pricing Strategy v2.0) |
