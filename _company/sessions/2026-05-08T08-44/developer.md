## 📋 MVP 기능 흐름 및 API/데이터 매핑 최종 안정성 검토 보고서

본 문서는 Designer가 제시한 UI/UX 명세서와 Supabase 데이터 스키마(`plans`, `user_subscriptions`)를 기반으로, 초기 개발에 필요한 핵심 사용자 흐름(User Flow)의 논리적 안정성과 백엔드 API 매핑의 정확성을 최종 검토하고 확인하는 것을 목적으로 합니다.

### 1. MVP 기능 흐름 (User Flow) 검토 및 안정성 확보

제시된 데이터 구조와 디자인 명세를 기반으로, 사용자가 목표를 달성하는 과정(User Flow)을 다음 세 단계로 정의하며 각 단계의 논리적 연결성을 검증합니다.

| 단계 | 사용자 액션 (UI/UX 기반) | 필요한 데이터 조회/생성 (DB 매핑) | 안정성 검토 결과 |
| :--- | :--- | :--- | :--- |
| **Step 1: 계획 선택** | 사용자는 `plans` 테이블에 정의된 구독 플랜(Starter, Accelerator, Scale)을 확인하고 선택한다. | `plans` 테이블 조회 (모든 플랜 정보 및 가격 구조). | **안정:** `plans` 테이블의 데이터가 UI/UX 명세서의 3-Tier 전략과 정확히 일치해야 함. 잘못된 매핑은 가격 책정 오류로 직결됨. |
| **Step 2: 구독 시작** | 사용자는 선택한 플랜에 대해 구독을 시작하는 CTA를 클릭한다. (인증 필요) | `user_subscriptions` 테이블에 새로운 레코드 삽입 및 사용자 ID 연결. | **안정:** RLS 정책이 이 단계에서 모든 접근 권한을 엄격히 통제해야 함. 인증된 사용자만 자신의 구독 상태를 업데이트할 수 있어야 함. |
| **Step 3: 상태 확인** | 사용자는 자신의 현재 구독 상태와 결제 기록을 확인할 수 있다. | `user_subscriptions` 테이블 조회 (현재 활성 플랜, 시작일, 상태). | **안정:** 데이터 일관성이 최우선이다. 프론트엔드에서 표시되는 정보는 반드시 백엔드(Supabase)의 원본 데이터와 1:1 매핑되어야 한다. |

**✅ 최종 안정성 요약:**
User Flow는 `plans` 테이블을 기준으로 동적으로 UI를 구성하고, `user_subscriptions` 테이블에 상태 변경을 기록하는 **선형적이고 단방향적인 흐름**으로 설계되었습니다. 이 흐름의 안정성은 Supabase RLS 정책이 데이터 무결성을 보장하는 데 전적으로 의존합니다.

### 2. API 엔드포인트 및 데이터 매핑 검토 (Next.js/Supabase 기준)

프론트엔드(Next.js)가 백엔드(Supabase)와 통신할 때 필요한 핵심 API 엔드포인트와 데이터 흐름을 확정합니다.

#### A. 필수 API 엔드포인트 정의

| 목적 | HTTP Method | Endpoint (Path) | 설명 | 보안 요구사항 |
| :--- | :--- | :--- | :--- | :--- |
| **플랜 조회** | `GET` | `/api/plans` | 모든 구독 플랜 정보(`plans` 테이블)를 가져옴. | 인증 필요 (RLS 검증) |
| **구독 생성** | `POST` | `/api/subscriptions` | 사용자별로 새로운 구독 기록을 삽입 (`user_subscriptions` 테이블). | 인증 필수 (사용자 소유 확인) |
| **내 상태 조회** | `GET` | `/api/user/subscription` | 현재 사용자의 활성 구독 상태를 조회. | 인증 필수 (자기 데이터 접근 검증) |

#### B. 데이터 매핑 상세 검토

1.  **`plans` 테이블 $\leftrightarrow$ 프론트엔드:**
    *   **매핑 확인:** `plans` 테이블의 각 행(Starter, Accelerator, Scale)은 Designer가 정의한 **컬러 시스템 및 타이포그래피(`Color_Accent_Vibrant`, H1 크기 등)**와 직접적으로 연결되어야 합니다.
    *   **안정성 확보:** 프론트엔드는 API 응답을 받아 단순 텍스트만 표시해서는 안 되며, 디자인 컴포넌트가 해당 데이터에 따라 **동적 스타일링(Dynamic Styling)**을 적용하도록 설계해야 합니다.

2.  **`user_subscriptions` 테이블 $\leftrightarrow$ 백엔드:**
    *   **매핑 확인:** 이 테이블은 사용자의 실시간 상태를 나타냅니다. `POST /api/subscriptions` 호출 시, 요청 본문(Body)에는 **사용자 ID와 선택된 플랜의 ID**만 포함되어야 합니다.
    *   **보안 및 성능:** 모든 데이터 삽입 시, Supabase의 RLS가 **`user_id` 필드가 현재 로그인한 사용자와 일치하는지**를 반드시 검증하도록 설정해야 합니다. 이는 무단 구독 생성(데이터 오염)을 원천 차단합니다.

### 3. 최종 권고 사항 (Developer Directive)

1.  **API 계약 명확화:** 위에서 정의된 세 가지 API 엔드포인트에 대해, **응답 데이터의 스키마(Response Schema)**를 즉시 확정하고 이를 프론트엔드 팀과 공유하십시오. 이는 향후 리팩터링 시 일관성을 보장합니다.
2.  **RLS 최종 점검:** `user_subscriptions` 테이블에 적용된 RLS 정책이 **`SELECT`, `INSERT`, `UPDATE` 작업 시 사용자 ID 기반의 제약 조건을 완벽히 구현했는지** 재확인하십시오. 이는 보안-by-디자인 원칙을 준수하는 핵심 단계입니다.
3.  **성능 최적화:** 데이터 조회 시, 불필요한 행(Row) 로드를 방지하기 위해 필요한 필드만 선택적으로 가져오는 **PostgREST/RLS 기반의 효율적인 쿼리 패턴**을 적용하여 Lighthouse 성능 목표(90+) 달성을 지원하십시오.
