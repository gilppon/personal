### 🛠️ API 엔드포인트별 테스트 케이스 (Critical Path Validation)

테스트 항목을 단순 기능 검증이 아닌, **'회사의 비즈니스 목표 달성 여부'**를 측정하는 관점에서 분류했습니다.

| 사용자 흐름 (User Journey) | 테스트 대상 엔드포인트 | 요청 데이터 예시 (Input Data) | 예상 성공 응답 (Expected Success Response) | 필수 검증 요소 (Validation Focus) |
| :--- | :--- | :--- | :--- | :--- |
| **1. 온보딩 및 인증** | `/api/auth/register`, `/api/auth/login` | 유효한 이메일, 비밀번호(최소/최대 길이 테스트 포함) | `{"success": true, "user_id": "..."}` | **[성공]** 권장비밀번호정책 적용 확인. **[실패]** 이미 존재하는 이메일 처리. |
| **2. 서비스 진입 및 상태 확인** | `/api/v1/subscription/status` | `auth.users`의 현재 User ID (인증 필요) | 구독 플랜 정보(`plan_id`, `expiry_date`, `is_active`) | **[핵심]** RLS 정책이 해당 사용자의 정확한 상태만 반환하는지 확인. |
| **3. 핵심 기능 호출 (가치 증명)** | `/api/v1/usage/check` | 특정 리소스 ID, 사용자 ID | `{"limit_used": 5, "limit_total": 10, "is_over_limit": false}` | **[핵심]** 사용량 기록(Increment)이 원자적 트랜잭션으로 처리되는지 검증. (동시성 오류 방지) |
| **4. 상위 플랜 구매/업그레이드** | `/api/v1/subscription/upgrade` | `plan_id`, 결제 토큰, 사용자 ID | 구독 정보 업데이트 및 영수증 기록 (`transaction_id`) | **[비즈니스]** 실제로 결제가 성공했을 때만 상태가 변경되고, 트랜잭션 로그가 남는지 확인. |
| **5. 권한 제한 접근 (Failure Path)** | `/api/v1/restricted-data` | 임의의 User ID 또는 과도한 요청량 | HTTP 403 Forbidden / 오류 메시지 (`{"error": "Insufficient privileges."}`) | **[보안]** 권한이 없는 사용자가 데이터를 가져가려 할 때, 백엔드에서 차단하는지 확인. (최악의 시나리오 테스트) |

### 🧪 Mock 데이터 사양 재정의 (Negative & Edge Case Focus)

Mock 데이터는 단순히 '작동되는' 케이스를 넘어, **시스템이 깨질 수 있는 경계값(Edge Case)**과 **비즈니스상 위험한 경우(Failure State)**를 커버하는 데 중점을 둡니다.

#### 1. 사용자 데이터 (`auth.users` / `user_profiles`)
*   **테스트 목적:** 권한 기반 접근 제어(RBAC) 테스트 및 이메일 인증 흐름 검증.
*   **필수 Mock 데이터 사양:**
    *   **[A] 정상 유료 사용자 (Golden Record):** 최신 플랜(`Accelerator`), 만료일이 충분히 남은 레코드. (성공 경로 기준점)
    *   **[B] 만료된 사용자 (Expired Record):** 구독 상태가 `inactive`이며, 사용 제한(Usage Limit) 카운터가 0인 레코드. (서비스 차단 흐름 테스트)
    *   **[C] 신규 가입자/테스트 계정:** 모든 플랜이 기본값으로 설정된 레코드. (온보딩 초기 상태 테스트)

#### 2. 구독 및 결제 데이터 (`user_subscriptions` / `transactions`)
*   **테스트 목적:** 금액 오류, 중복 구매 방지 등 금융 로직의 무결성 검증.
*   **필수 Mock 데이터 사양:**
    *   **[D] 부분 환불/취소 시나리오:** 구독 기간이 남아있으나 일부만 사용하고 취소된 경우 (Partial Refund). 이 경우 `refund_amount`와 `status: refunded`가 기록되어야 함.
    *   **[E] 동시성 오류 발생 Mock:** A 사용자가 동시에 두 번의 업그레이드 API를 호출했을 때, 시스템이 어느 하나의 트랜잭션을 거부하고 에러 코드를 반환하는지 테스트할 수 있는 락(Lock) 관련 데이터 구조 시뮬레이션.

#### 3. 사용량 및 제한 데이터 (`usage_log`)
*   **테스트 목적:** 카운터 오버플로우, 동시 요청에 의한 누적 오류 방지.
*   **필수 Mock 데이터 사양:**
    *   **[F] 최대치 도달 직전:** `limit_used`가 `limit_total - 1`인 상태의 로그 (다음 호출에서 경고/차단이 발생해야 함).
    *   **[G] 초기화된 사용량:** 구독 플랜 변경 시, 모든 `usage_log` 레코드가 성공적으로 삭제되거나 아카이브되는 과정 테스트 데이터.

### 💡 Researcher의 분석 및 인사이트 (So What?)

1.  **데이터 구조 최우선 검증 (Risk Priority):** 현재 MVP 단계에서 가장 큰 위험은 **'사용량(Usage)' 트랜잭션의 동시성 오류**입니다. 사용자가 동시에 기능을 호출했을 때, 카운터가 오버카운트되거나 기록되지 않아 서비스 과금이 발생하거나 기능이 무한정 사용할 수 있게 되는 치명적인 데이터 무결성 문제가 발생할 수 있습니다.
    *   ✅ **[Actionable Suggestion]**: 개발팀은 `usage_log` 쓰기(Write) API에 트랜잭션 격리 수준(Isolation Level)을 최고 수준으로 설정하고, 해당 로직을 가장 먼저 검증해야 합니다.

2.  **Mock 데이터의 목적 변화:** Mock 데이터를 단순히 '데이터가 있음'으로 준비하는 것이 아니라, **시스템이 실패했을 때의 모습을 미리 정의**하여 테스트에 활용해야 합니다. (예: `expired record`, `partial refund` 등) 이는 개발 초기 단계에서 회사의 비즈니스 로직을 가장 안전하게 보호할 수 있는 방법입니다.
